# 研发立项评审

由 Patsnap Eureka 提供。

当决策对象为具体项目、提案、创新包或立项文件集时，使用本技能。
目标是判断项目是否应推进、与公开现有技术的相似程度、
哪些创新主张仍然成立，以及在进入下一关口前还缺少哪些证据或材料。

客户可见价值应首先落在以下四个问题上：

1. 该项目是否值得推进？
2. 与公开现有技术的接近程度如何？
3. 创新主张是否站得住脚？
4. 进入下一关口前还缺少什么？

## 适用场景

- 用户提供项目或提案，询问是否值得推进
- 具体项目的正式立项评审报告
- 预审 / 公开新颖性预查 / 面向客户的预筛查
- 围绕提案的通过/否决或预算释放建议
- 项目级新颖性 / 可行性 / 趋势 / 风险评审
- 逐条创新点评估
- 附证据包的管理层评审材料集

## 不适用场景

- 无具体项目对象的纯技术路线比较 → 路由至 `tech-route-comparison`
- 按主题的公司技术画像或竞争格局分析 → 路由至 `company-tech-profile` 或 `competitive-landscape`
- 法律专利意见、FTO（自由实施）、侵权或诉讼分析
- 从零起草原始提案（本技能负责评审，不负责撰写）
- 无项目级评审深度的通用公司概览

## 评审模式

| 模式 | 适用场景 | 典型输出 |
|------|----------|----------|
| `screen`（预筛查） | 预审、早期评审、面向客户的售前 | 快速通过/否决 + 新颖性边界 + 关键缺口（6-8 页） |
| `review`（标准评审） | 标准立项评审 | 完整评审报告 + 新颖性说明 + 证据包（10-14 页） |
| `innovation`（创新点评审） | 创新点深度评审 | 逐点创新评估 + 差异化对比表（8-12 页） |
| `assurance`（专家评审） | 专家评审、委员会级材料包 | 深度证据充分性审计 + 评级逻辑 + 阶段关口矩阵（12-18 页） |

默认模式选择：

- 若请求为预审、新颖性预查、重叠检查、"是否值得推进"或早期面向客户的评审 → 默认 `screen`
- 若请求为专家评审、通过/否决、预算释放、证据充分性或委员会级材料包 → 默认 `assurance`
- 其他情况 → 默认 `review`

## 核心原则

### 提案作为输入，而非事实

提案是提取信息的起点，不是已核实事实的来源。
提案中的主张不能成为评审结论，除非经过外部佐证或明确标注为"仅来自提案"。

### 四层证据分离

评审中每一条实质性主张必须标注为以下之一：

1. **提案陈述**：提案所述内容（引用或摘要）
2. **外部佐证事实**：经专利、论文或官方来源确认
3. **有据推断**：基于多个信号的合理结论
4. **开放缺口**：证据不足——说明缺少什么以及如何补充

不得让提案陈述的主张悄然成为评审结论。

### 新颖性作为一级可见模块

新颖性检索不是基线检索的副产品。评审必须明确说明：

- 新颖性检索范围（数据库、时间窗口、领域范围）
- 主要对比对象（现有技术、现有路线、已有方案）
- 逐点重叠评估
- 残余差异点
- 每项主要主张的明确新颖性判断等级
- 检索局限性及未检索区域

当新颖性因现有技术而降级时，构建结构化差异化边界表，
展示重叠程度、残余差异点及替代风险。

### 材料完整性作为一级模块

即使平台无法解决客户的完整数据治理问题，
报告也应明确说明哪些缺失材料或数据当前正在阻碍：

- 新颖性或重叠边界判断
- 可行性判断
- 下一步立项行动

若内部档案或非公开项目数据库不可用，应明确说明。
不得暗示仅凭公开检索即可证明内部重复项目排查的完整性。

## 工具路由与降级策略

本技能适用于多种工具环境。检索前，检测可用能力并选择可达的最高层级。

### 第 0 层（始终优先）：提案材料

- 用户提供的提案、立项报告、创新包
- 在任何外部检索之前，提取项目摘要、研究主题、创新点、承诺产出及提案陈述主张
- 证据等级：提案陈述（未经外部核实）

### 第 1 层（推荐）：结构化专利/论文检索

- 使用宿主环境最佳的结构化专利和论文检索栈，围绕归一化创新点测试新颖性、可行性和前沿证据。
- 通常指结构化检索加记录级深度获取。
- 证据等级：S/A

### 第 2 层（降级）：Exa 检索 + 配套技能

- 使用宿主环境最佳的广域网络研究工具，以及可用的学术或财务配套通道。
- Exa、Tavily、Brave、官方申报文件及领域专项研究数据库均为示例，非硬性要求。
- 证据等级：A/B
- 相比第 1 层的覆盖损失：无结构化专利字段检索，无受让人过滤，新颖性结论精度较低

### 第 3 层（降级）：通用网络工具

- 使用宿主环境中可用的通用网络检索和页面/PDF 阅读工具。
- 证据等级：B/C

### 第 4 层（最低限度）：纯 LLM + 提案材料

- 无需外部工具
- 证据等级：C/U
- 评审仍可提取和整理提案主张、识别逻辑缺口，并标注需要哪些外部证据

### 路由规则

- 无论工具是否可用，始终从提案材料（第 0 层）出发。
- 选择可用的最高外部层级进行核实。
- 当某层级不可用时，明确说明降级情况。
- 不得将仅凭公开检索等同于完整的重复项目排查。
- 使用第 3 层或第 4 层时，增加明确的缺口说明，降低新颖性和完整性主张的置信度。

## 默认工作流

### 第 0 步：冻结范围

在任何检索前确认或推断以下内容：

- `project_name`：具体项目或提案对象
- `topic`：研究主题（若未说明，从材料中推断）
- `decision_gate`：预批准 / 通过-否决 / 预算释放 / 专家评审
- `mode`：screen / review / innovation / assurance
- `deliverable`：简报 / 报告 / docx / pdf / 证据包
- `audience`：客户 / 内部 / 委员会 / 管理层
- `time_window`：默认近 3-5 年
- `review_focus`：新颖性 / 可行性 / 趋势 / 风险 / 评级（可多选）
- `innovation_scope`：从材料中提取或由用户指定的创新点

若用户提供了材料但未说明研究主题，从材料中推断暂定主题并记录为暂定。

若用户完全未提供具体项目对象，不强行使用本技能——
重定向至 `tech-route-comparison` 或 `company-tech-profile`。

### 第 1 步：提取提案材料

在任何外部检索之前：

1. 阅读提案并提取：
   - 项目摘要和研究主题
   - 陈述的创新点（归一化为简短对比集）
   - 承诺的产出和交付物
   - 提案关于新颖性、可行性和市场的陈述主张
   - 技术路径描述（主链、依赖关系、测量方法）
2. 构建创新登记册：每个创新点作为原子条目，保留提案原始表述。
3. 识别提案未涉及的内容：缺失数据、未陈述假设、逻辑缺口。
4. 此阶段将所有内容标注为 `提案陈述`。

### 第 2 步：检索外部证据

1. 围绕每个归一化创新点检索现有技术和现有路线。
   - 第 1 层：围绕每个创新点进行结构化专利/论文检索
   - 第 2 层：网络研究加学术配套检索
   - 第 3/4 层：通用网络或仅基于提案的推理
2. 对每个创新点进行专项新颖性检查：
   - 检索范围：主题领域内的专利 + 论文 + 标准
   - 记录：数据库/来源范围、时间过滤、字段范围、关联新颖性点、筛选规则
   - 识别：最接近的现有技术、重叠程度、残余差异点
3. 检索可行性和前沿支撑：
   - 提案的技术方法是否在其他地方得到验证？
   - 当前技术水平如何？
   - 是否存在已知失效模式或阻碍因素？
4. 在相关时检索趋势和政策背景：
   - 行业方向、监管环境、标准状态
   - 仅使用网络来源进行当前状态主张（标准、法规、上市状态、产品化证明）

### 第 3 步：按模式构建分析

根据当前模式选择模块：

| 模块 | screen | review | innovation | assurance |
|------|--------|--------|------------|-----------|
| 提案提取 | ✓ | ✓ | ✓ | ✓ |
| 新颖性边界 | ✓ | ✓ | ✓ | ✓ |
| 创新点评估 | 简要 | 标准 | 深度 | 深度 |
| 可行性检查 | 简要 | 标准 | 简要 | 深度 |
| 趋势/前沿背景 | 可选 | 标准 | 可选 | 标准 |
| 材料完整性 | ✓ | ✓ | ✓ | ✓ |
| 评分/评级 | 跳过 | 可选 | 可选 | 必须 |
| 阶段关口矩阵 | 跳过 | 跳过 | 跳过 | 必须 |
| 反证审查 | 跳过 | 可选 | 可选 | 必须 |
| 治理审查 | 跳过 | 跳过 | 跳过 | 可选 |

### 第 4 步：综合与起草

按照适合当前模式的输出框架撰写报告。

对于面向客户或售前交付物，按以下顺序将可见价值置于首页：

1. 建议结论
2. 新颖性与重叠边界
3. 关键问题与风险
4. 缺失材料、数据及下一步行动

不得让深度技术分解在主叙述中遮盖这些问题。

## 输出框架

### 预筛查模式（Screen）

1. 执行建议（推进 / 附条件推进 / 暂缓）
2. 新颖性边界摘要（1-2 段）
3. 关键风险与阻碍因素
4. 缺失材料与下一步行动

### 标准评审模式（Review）

1. 执行建议
2. 项目范围与提案陈述主张
3. 新颖性与重叠判断（附对比表）
4. 可行性与证据充分性
5. 主要风险与材料缺口
6. 建议与条件

### 创新点评审模式（Innovation）

1. 创新评估执行摘要
2. 逐点创新分析（每点：提案主张 → 外部证据 → 重叠 → 残余差异点 → 判断）
3. 差异化边界表
4. 创新主张的风险
5. 建议

### 专家评审模式（Assurance）

1. 附置信度的执行建议
2. 项目范围与提案连贯性评估
3. 新颖性与重叠（附详细对比附件）
4. 可行性、证据充分性与技术分解
5. 评分与评级逻辑（附明确评分标准）
6. 阶段关口矩阵（条件、责任人、时间线、不满足后果）
7. 材料缺口与数据就绪度
8. 反证与弱化条件

每条主张必须引用其来源类型和标识符，例如：`[专利: CN1234567B]`、
`[论文: DOI 或标题]`、`[网络: 来源名称]`、`[提案: 章节/页码]`。

## 正式输出规范

对于大型企业、公共部门机构和科研机构等正式评审受众：

1. 删除空洞的开场铺垫，如趋势引言或时代背景铺陈。
2. 删除宣传性语言，以具体评审判断替代。
3. 以具名证据和明确引用替代模糊归因。
4. 将程式化的对仗表述改写为直接的独立判断。
5. 不得将控制层术语（如 `proposal-only`、`prior-art`、`assurance`、`gate`、`owner`）
   泄漏至主报告，除非该术语必要且有明确解释。

标题和正文应使用适合书面评审或委员会备忘录的简洁正式措辞。

当新颖性是现实决策问题时，将方法和逐点对比提升至 `novelty-note.md`。
合适的文件标题为"公开文献与专利预查新说明"或"查新与对标附件"。

## 完成关口

### 所有模式

- [ ] 已识别具体项目对象（不仅是主题）
- [ ] 在外部检索前已提取提案陈述主张
- [ ] 全程维持四层证据分离
- [ ] 已明确说明工具层级和覆盖局限性
- [ ] 已进行材料完整性评估
- [ ] 若内部档案不可用，重复项目边界标注为未核验

### 预筛查模式附加项

- [ ] 附条件的通过/否决建议已呈现
- [ ] 新颖性边界摘要已呈现
- [ ] 已识别关键缺口和下一步行动

### 标准评审模式附加项

- [ ] 每个创新点至少执行 2 次专利检索和 1 次论文检索
- [ ] 新颖性对比表已呈现
- [ ] 每条主要主张均有可追溯的来源引用

### 创新点评审模式附加项

- [ ] 每个创新点已单独评估
- [ ] 差异化边界表已呈现
- [ ] 每点的残余差异点已明确说明

### 专家评审模式附加项

- [ ] 评分标准明确且可追溯
- [ ] 阶段关口矩阵含条件、责任人和后果
- [ ] 反证审查已完成
- [ ] 技术分解涵盖主链、依赖关系和失效模式

## 护栏

- 提案是输入对象，不是事实来源。不得让提案措辞替代外部证据。
- 不得暗示仅凭公开检索即可证明完整覆盖或内部重复项目排查。
- 不得让原始检索命中数或内部台账术语成为面向管理层的主要论据。
- 将提案承诺、路线图主张和商业化陈述视为信号，直至得到佐证。
- 将评分作为支撑层，而非主要卖点。在主体中优先使用直接判断（推进 / 附条件推进 / 暂缓），而非密集的加权评分表。
- 当新颖性降级时，不要止步于文字警示——构建结构化差异化边界表。
- 当证据基础薄弱时，不要强行给出明确的通过/否决——优先给出附明确后续证据收集步骤的条件性建议。
- 将新颖性方法（范围、查询、局限性）与新颖性结论分开。
- 不得在未明确标注的情况下混淆提案陈述主张与外部核实事实。
- 正式报告中，不得将控制层术语泄漏至面向管理层的叙述中。
- 将内部重复项目排查视为数据依赖项——当内部项目历史不可用时，不得承诺其完整性。
