# 能源化工材料催化剂制备与评价方案审核 Skill

## 1. 使用边界

本 Skill 面向催化剂制备与评价方案的前置审核。核心问题不是“方案写得是否漂亮”，而是：

1. 实验人员是否能够照着材料完成制备；
2. 关键条件是否足以复现实验；
3. 样品、变量、对照之间是否能够支持归因；
4. 评价条件是否足以形成可比结果；
5. 性能或机理主张是否配置了对应验证项目。

本 Skill 不做实验结果真实性判断，不做专利壁垒分析，不做工业放大设计，不替代催化剂专家最终判断。

## 2. 输入材料处理原则

用户可能只上传一个 Word、PDF、图片截图或一段文字。不得要求用户先整理成标准目录。必须先识别材料性质，再选择审核模式。

材料类型与处理方式：

| 材料类型 | 处理方式 |
|---|---|
| 制备步骤型方案 | 执行完整审核，重点检查步骤、条件、样品与对照 |
| 制备与评价完整方案 | 执行完整审核，检查制备、评价、对照、验证路径 |
| 研发设想或智能体初稿 | 执行草案级审核，重点输出缺失项和补充建议 |
| 论文 Methods 片段 | 执行方法复核，重点检查可复现性和条件完整性 |
| 整篇论文 | 先定位 Methods / Catalyst preparation / Catalytic testing，再审核相关部分 |
| 专利实施例 | 执行实施例方案化审核，说明其不是标准研发 proposal |
| 信息过少 | 输出材料不足提示，不得强行给出完整审核结论 |

如果输入是图片或扫描件，必须先忠实转写。转写时保留原文单位、数字、温度、时间、气氛、样品名、化学式，不得补写缺失信息。

## 3. 强制执行流程

必须按以下顺序执行：

1. 读取输入材料；若为截图或扫描件，按 `prompts/00_faithful_transcription.md` 完成忠实转写。
2. 依据 `rules/material_type_rules.yaml` 判断材料性质和审核模式。
3. 抽取方案要素：制备步骤、样品设计、对照关系、评价条件、表征计划、性能或机理主张。
4. 调用唯一入口脚本：

```bash
python scripts/run_audit.py --input <input_file_or_transcription> --out outputs
```

5. 脚本根据固定规则生成 `report_context.json`、HTML 报告和 Word 报告。
6. 调用校验逻辑，确认 HTML 和 Word 均存在、内容非空、问题统计一致、文件名固定。

不得临时创建 `fix_json.py`、`rebuild_json.py` 等补救脚本。不得直接手写 HTML 或 Word 正文绕过 `run_audit.py`。


## 3.1 运行环境与输出约束

本 Skill 不得在审核过程中执行 `pip install`，不得请求用户确认安装运行环境。DOCX 读取、DOCX 生成和输出校验均应使用 Python 标准库完成。

如果输入文件无法读取，必须直接停止并说明原因，不得生成半成品报告。每次运行前必须清空 `outputs/` 目录，所有输出只能写入 `outputs/`。不得在 Skill 根目录生成 `report_context.json`、HTML 或 Word 报告。

执行结束后，聊天区只输出简短完成提示，不得复述完整报告正文。完整审核内容只以 HTML 和 Word 报告为准。

## 4. 报告结构

最终报告标题固定为：

`催化剂制备与评价方案审核报告`

报告固定包含：

1. 审核对象与材料性质
2. 总体审核意见
3. 重点问题清单
4. 分项审核结果
   - 制备步骤可执行性
   - 变量设计与归因关系
   - 对照样与基准样设置
   - 评价条件与数据可靠性
   - 性能主张与验证项目对应关系
5. 制备步骤核对表
6. 样品与变量关系核对表
7. 修改建议
8. 附录：方案要素提取结果、审核规则与版本信息

HTML 可以不展示完整附录；Word 必须包含附录。

## 5. 正式用词要求

报告面向科研人员、项目负责人和专家，不得使用产品化、口语化或 AI 味表达。

禁止使用：
- 红旗清单
- Dashboard
- AI 审核
- Agent 方案
- 打脸
- 牛逼
- 风险雷达

推荐使用：
- 重点问题清单
- 审核结论摘要
- 分项审核结果
- 方案材料
- 待审方案
- 当前设计不足以支撑该主张
- 需补充事项

## 6. 判定等级

内部可以使用枚举，报告中显示中文：

| 内部等级 | 报告显示 |
|---|---|
| SUFFICIENT | 设计较充分 |
| PARTIAL | 设计基本具备，但需补充完善 |
| INSUFFICIENT | 当前设计不足以支撑要求 |
| NOT_APPLICABLE | 不属于本次审核范围 |

问题级别：

| 内部等级 | 报告显示 |
|---|---|
| HIGH | 关键问题 |
| MEDIUM | 重要问题 |
| LOW | 一般问题 |

## 7. 稳定性要求

同一输入文件、同一 Skill 版本、同一规则版本、同一转写文本下：

1. 核心问题清单必须一致；
2. 问题级别必须一致；
3. 问题统计必须一致；
4. HTML 与 Word 的核心结论必须一致；
5. 输出文件名和目录必须一致。

如输入为图片，转写文本不同会导致审核结果不同。因此执行者必须优先保证转写一致性。


## 8. 样品与工序识别要求

样品名称识别必须排除气氛、溶剂、设备、溶液编号和纯条件字段。例如 `H2/Ar`、`20% H2/Ar`、`空气`、`无水乙醇`、`去离子水`、`马弗炉`、`管式炉` 不得作为样品进入“样品与变量关系核对表”。

制备步骤识别应优先归类为正式工序名称，包括：前驱体溶解、混合/搅拌、水热处理、固液分离/洗涤、干燥、研磨、煅烧/焙烧、浸渍、蒸发/老化、还原、样品命名、对照样制备。不得只以“搅拌”“标记”等泛化动词替代关键工序。
