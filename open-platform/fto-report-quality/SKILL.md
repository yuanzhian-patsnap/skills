# FTO报告质量审核技能（fto-review）

通用版 · 适用于任意公司 / 法务团队 / 专利代理所 · 标准化 · 系统化 · 可追溯 · 可降级  
**版本**：v9.0（2026-05-27 矩阵场景+五轨检索+一票否决+摘要优先+Jackknife召回+桑基图+优先级建议+Harness逻辑校验）  

## 核心升级（v9.0 vs v8.1）

| 升级点 | v8.1 | v9.0 |
|--------|------|------|
| 场景识别 | 通用/展会/行业三类 | 市场维度×用途维度矩阵，评分权重动态调整 |
| 检索轨道 | 四轨 | 五轨（新增时序验证轨，覆盖近2年审中专利） |
| 召回率估算 | Chapman双样本 | Jackknife多样本+轨间重叠修正系数 |
| 致命缺陷处理 | 总分<60才不合格 | 一票否决机制，直接触发不可用警告 |
| 报告输出模式 | 固定章节顺序 | 新增摘要优先模式（高管/法务读者） |
| 召回率可视化 | 简单仪表盘 | 桑基图展示四轨命中重叠 |
| 风险建议排序 | 高/中风险分类 | 可执行难度×风险消除幅度矩阵排序 |
| Harness校验 | 结构/样式/禁用表述 | 新增跨字段逻辑一致性规则 |
| D4分值 | 3分 | 6分，与F5联动扣分 |
| B1/B2权重 | 固定 | 标准化产品场景下B2上浮、B1下调 |

---

**五轨检索**：
1. **语义轨道**：`patsnap_search`（`search_strategy=["semantic"]`，topk可配）
2. **关键词轨道**：`patsnap_search`（`search_strategy=["keyword","filter"]`，含地域过滤）
3. **IPC分类轨道**：从前两轨自动提取 top-5 IPC 再检索
4. **申请人扩展轨道**：从前两轨提取主要申请人再检索
5. **时序验证轨道**：专门检索近2年新申请（含审中状态），捕捉即将授权高风险专利

**Jackknife召回率估算**：五轨合并去重后与原报告专利池做比对，执行 Jackknife 多样本估算并对轨间重叠比例做修正系数，输出原报告召回率 R_A 及召回率评级。  
**输出模式**：默认只生成并交付 `.html` 报告（v7.1 HTML 内嵌固定 CSS，布局和类名固定，仅内容与状态 class 变化）。  
**双模块结构**：本 skill 内置两个模块，但默认只执行报告生成模块。
- `scripts/generate_report.py`：生成 FTO 报告质量审核 HTML。
- `scripts/validate_report.py`：内置 harness，可选后置检查已生成 HTML 的结构、样式、逻辑一致性、禁用表述及跨字段逻辑一致性。

**强制执行规则（非常重要）**：
- 只要用户要求使用本 skill 审核 FTO 报告质量，最终必须交付 `.html` 文件，不得只在对话中输出 Markdown、文字摘要或口头结论。
- 生成 HTML 时必须调用 `scripts/generate_report.py`；即使报告内容尚未完全解析，也要先生成带固定章节的 HTML 骨架，并在缺失项处标注"原报告未提供/未验证/待补充"。
- 最终回复必须给出生成的 HTML 文件路径。
- 不要把 harness 作为生成 HTML 的前置条件；harness 只在用户明确要求校验时运行。

**HTML/CSS稳定性硬约束**：
- 生成报告必须调用 `scripts/generate_report.py`，不得由模型临场手写 HTML。
- HTML 章节顺序、标题文本、表格列、CSS class 名称必须保持固定。
- CSS 唯一来源为 `assets/fto_report.css`，生成时原样嵌入 `<style id="fto-fixed-css" data-version="v7.1-fixed">`。
- 禁止输出 `style=` 内联样式；所有视觉状态只能使用固定 class：`grade-*`、`recall-*`、`risk-*`、`pct-*`。
- 内容缺失时填入"原报告未提供/未验证/N/A"，不得删除章节或改变布局。

**内置Harness可选检查约束**：
- 生成报告时不得自动执行 harness，不得生成候选 HTML 作为发布门禁。
- harness 结果不得影响 `fto_review_report_YYYYMMDD.html` 的正常生成。
- 只有用户明确要求"使用 harness 检查/校验/质检报告"时，才调用 `scripts/validate_report.py` 检查已生成的正式 HTML。
- harness 检查只输出独立的 JSON/HTML 校验结果，不自动改写 FTO 审核报告。
- harness 检查结果为 `fail` 时，应把失败项作为整改建议反馈给用户；不得为了通过校验自动改写侵权结论、专利风险等级或 FTO 可实施性等实质法律判断。
- v9.0新增：harness 跨字段逻辑一致性校验规则，见下文"Harness跨字段校验规则"章节。
- 详细检查项见 `references/harness-checks.md`；视觉样式见 `assets/harness_report.css`。

## 技能定位与触发边界

**使用本技能的场景**：
- 用户上传或提交了一份已有的FTO报告或知识产权风险自评报告，要求进行质量审核
- 任意公司新产品/新技术上市前的FTO报告验收
- 重大招投标前的专利风险报告审核
- **展会参展前的知识产权风险自评报告审核**（任意国家/地区展会）
- 海外市场拓展前的知识产权风险评估验收
- 技术引进/合作前的专利尽职调查报告审核
- 年度知识产权合规检查
- 专利代理所为客户出具FTO报告后的内部质量自查

**不使用本技能的场景**：
- 从零开始开展FTO分析 → 使用 `fto-analysis`（如已安装）或直接委托专利团队
- 单一专利新颖性分析 → 使用 `novelty-check`
- 多文件创造性分析 → 使用 `non-obviousness-check`

| 信息项 | 自动提取方式 | 无法提取时的询问话术 |

| **行业/产品领域** | 从报告技术描述提取 | "请简要描述产品所属行业，以便启用行业特色审核要点。" |

> ⚠️ **原则**：能从报告中提取的信息不重复询问；未提及则使用合理默认值并在审核报告中注明。

## 场景识别（v9.0 矩阵式升级）

> v9.0 将场景识别从三分类升级为**市场维度 × 用途维度**二维矩阵，评分权重随场景组合动态调整。

### 场景矩阵定义

| 市场维度 \ 用途维度 | 产品上市 | 展会参展 | 出口/海外拓展 | 招投标 | 技术引进/许可 | 内部合规 |
|-------------------|---------|---------|-------------|-------|------------|---------|
| 中国大陆 | CN-Launch | CN-Expo | — | CN-Bid | CN-License | CN-Compliance |
| 欧美市场 | US/EU-Launch | US/EU-Expo | US/EU-Export | US/EU-Bid | US/EU-License | — |
| 东南亚/新兴市场 | SEA-Launch | SEA-Expo | SEA-Export | — | SEA-License | — |
| 全球/多市场 | Global-Launch | Global-Expo | Global-Export | — | Global-License | — |

### 场景自适应权重规则

**标准化产品场景**（CN-Launch / US/EU-Launch / Global-Launch 且产品标准化程度高）：
- B2（侵权风险判定）权重上浮至 10 分，B1（权利要求逐项对比）权重下调至 8 分
- A1（数据库覆盖）权重上浮至 10 分（标准化产品漏检后果更严重）

**展会场景**（*-Expo）：
- C3（风险缓释建议）权重上浮至 8 分（展会现场处置时效性要求高）
- D4（版本控制规范）联动加权（展品版本混乱高发场景）

**跨境出口/海外拓展场景**（*-Export / US/EU-* / Global-*）：
- B4（有效性分析）权重上浮至 8 分（境外专利有效性核查难度更高）
- A3（时间范围）权重上浮至 7 分（境外专利审查周期长，时序风险高）

> 评分权重调整时，必须在报告中注明"已启用[场景名称]自适应权重，本次满分仍为100分（各维度内部重新比例分配）"。

### 场景识别优先级
1. 用户明确指定的审核场景优先。
2. 报告标题、摘要、目标市场、用途中出现"展会/参展/展品/展台/现场展示/禁令/海关扣押"等信息时，启用展会专项。
3. 报告仅涉及上市、出口、招投标、技术引进、内部质量自查等场景时，使用通用FTO质量标准；展会专项列为 N/A，不扣分。
4. 识别到明确行业时，可启用行业特色检查；行业要求必须来自报告、用户输入或已配置的行业标准，不得臆造。

## 致命缺陷一票否决机制（v9.0 新增）

> 无论综合得分多高，触发任一致命缺陷条件即直接输出**不可用警告**，报告质量等级强制降为"不可用（Fatal）"。

### 致命缺陷触发条件

| 编号 | 致命缺陷描述 | 对应审核项 |
|------|------------|---------|
| F-01 | 目标市场完全遗漏（报告未明确任何目标市场/司法管辖区） | F1 |
| F-02 | 权利要求文本版本错误（使用已修改/无效版本的权利要求文本进行对比） | F2 / B1 |
| F-03 | 核心产品/技术完全未覆盖（检索主题与产品实际技术特征无关联） | A2 / B1 |
| F-04 | 法律状态截止日缺失且报告超过18个月未更新 | F3 |
| F-05 | 高风险专利有效性完全未核查 | B4 |
| F-06 | 无任何风险缓释建议且报告包含高风险专利 | C3 |

### 一票否决输出规则
- 触发致命缺陷后，必须在报告最顶部显示红色警告横幅：`⛔ 致命缺陷警告·报告不可用于决策`
- 列明触发的具体缺陷编号与描述
- 综合质量等级强制标注为 `Fatal`（超出原评分体系）
- 仍必须生成完整 HTML 报告（不因致命缺陷中止生成），骨架章节照常输出，缺失内容标"致命缺陷·待补充"
- 致命缺陷不因其他维度高分而抵消

## 三层审核方法论

### 第一层：事实层审核（Evidence & Reproducibility）
核查报告中的事实基础是否足以支撑后续判断：目标产品明确性、目标市场明确性、检索可复现性、专利池完整性、专利类型区分。

### 第二层：法律判断层审核（Legal Reasoning）
核查报告的侵权和有效性判断链条是否成立：权利要求解释、侵权类型区分、有效性稳定性核查、结论置信度。

### 第三层：决策可用层审核（Actionability）
核查报告能否支持真实业务决策：决策问题回答、处置路径闭环、风险边界声明、后续监控、管理层摘要。

## 独立验证模块 v9.0（五轨检索 + Jackknife召回率）

### 五轨检索流程

**轨道1：语义轨道**
```
调用 patsnap_search(
  search_strategy=["semantic"],
  semantic_query=<产品技术描述>,
  sources=["patent"],
  filters={jurisdiction: <目标市场>},
  topk=50
)
```

**轨道2：关键词轨道**
```
调用 patsnap_search(
  search_strategy=["keyword","filter"],
  keywords=<从产品描述提取的核心技术词>,
  sources=["patent"],
  filters={jurisdiction: <目标市场>, date_from: <有效期内>},
  topk=50
)
```

**轨道3：IPC分类轨道**
```
从轨道1+2结果自动提取 top-5 IPC 大组
调用 patsnap_search(
  search_strategy=["keyword","filter"],
  keywords=<核心关键词>,
  filters={ipc: <top-5 IPC>, jurisdiction: <目标市场>},
  topk=30
)
```

**轨道4：申请人扩展轨道**
```
从轨道1+2结果提取 top-5 申请人（含竞争对手）
调用 patsnap_search(
  search_strategy=["keyword","filter"],
  keywords=<核心关键词>,
  filters={assignees: <top-5申请人>, jurisdiction: <目标市场>},
  topk=30
)
```

**轨道5：时序验证轨道（v9.0新增）**
```
调用 patsnap_search(
  search_strategy=["keyword","filter"],
  keywords=<核心关键词>,
  sources=["patent"],
  filters={
    jurisdiction: <目标市场>,
    legal_status: ["pending"],
    date_from: <当前日期-730天>,
    date_to: <当前日期>,
    date_type: "application"
  },
  topk=30
)
```
> 轨道5专门捕捉近2年新申请、法律状态为"审中"但技术特征高度相关的专利，识别即将授权的潜在高风险。时序验证结果单独列示，不纳入主召回率计算，作为"待授权风险预警"独立模块输出。

### Jackknife多样本召回率估算（v9.0升级）

```
设：
  A = 原报告专利池（n_A）
  B = 五轨合并去重后独立专利池（n_B）
  C = A ∩ B 重叠（n_C）

步骤1 - 轨间重叠修正：
  计算各轨道两两重叠比例 ρ_ij
  修正独立性系数 κ = 1 - mean(ρ_ij) （κ ∈ [0,1]）
  有效独立样本量 n_B_eff = n_B × κ

步骤2 - Jackknife估算：
  将独立池 B 分为 k=5 折，每折移除后重新估算总量 N̂_(-i)
  N̂_jackknife = k × N̂_full - (k-1) × mean(N̂_(-i))

步骤3 - 召回率与置信区间：
  R_A = n_A / N̂_jackknife
  SE(R_A) = std(N̂_(-i)) / N̂_jackknife²  （近似）
  95% CI：R_A ± 1.96 × SE(R_A)
```

**召回率评级**：≥85% 充分✅ / 70-84% 偏低⚠️ / <70% 严重不足❌  
**修正提示**：若 κ < 0.5（轨间重叠过高，独立性不足），标注"⚠️ 检索轨道独立性不足，召回率估算置信度降低，建议扩充差异化检索策略"。

> ⚠️ 只有在获得真实独立专利池 B 且可与原报告专利池 A 比对时，才允许计算 Jackknife 估算召回率。

### 降级方案（工具不可用时）

| 情形 | 降级方案 |
|------|----|
| `patsnap_search` 不可用 | 标注 `[未完成独立检索验证·缺少可用检索工具]`，不计算召回率，不声明存在或不存在漏检 |
| 原报告无专利号列表 | 仅输出独立检索结果，召回率字段填"无法比对" |
| topk结果过少（<10件） | 标注"独立检索结果不足，召回率估算置信度低" |
| 轨道5时序轨不可用 | 标注"时序验证轨未执行，近2年审中专利未覆盖"，不影响主召回率 |

## 六步标准化审核流程

### 步骤1：解析报告 & 锁定审核范围
读取报告全文，提取关键信息，执行矩阵式场景识别，确定场景组合代码（如 CN-Launch），建立三层审核矩阵，确认场景自适应权重方案。

### 步骤2：检索策略质量审核（25分，场景自适应）

| 子项 | 标准分 | 标准化产品场景 | 展会场景 | 跨境出口场景 |
|------|--------|-------------|---------|-----------|
| A1 数据库覆盖 | 8分 | **10分** | 8分 | 8分 |
| A2 检索关键词策略 | 6分 | 6分 | 6分 | 6分 |
| A3 时间范围 | 5分 | 5分 | 5分 | **7分** |
| A4 策略透明性 | 6分 | 6分 | 6分 | 6分 |

> 场景自适应时各维度内部重新比例分配，满分仍为25分。

**事实层必查项（F1~F5）**：目标市场锁定（F-01关联）、权利要求文本版本（F-02关联）、法律状态核查日（F-04关联）、专利族口径、检索可复现性（F5，与D4联动扣分）。

### 步骤3：独立检索验证与比对（核心差异化步骤）

#### 3.1 五轨独立检索
按上述五轨检索流程执行。轨道1-4合并去重，执行 Jackknife 估算。轨道5（时序验证）单独输出"待授权风险预警"清单，不纳入主召回率。

#### 3.2 高风险专利有效性验证
- Quick模式：前2件高风险专利（+ 遗漏专利中前1件高风险）
- Deep模式：全部高风险专利（含遗漏）
- 优先调用 `novelty-check` / `non-obviousness-check`；不可用时降级为 `patsnap_search` 人工比对

**差异标注体系**：

| 标注 | 含义 |
|------|------|
| `[独立验证发现·严重]` | 遗漏高风险有效专利 |
| `[独立验证发现·重要]` | 遗漏中风险有效专利 |
| `[独立验证发现·召回不足]` | 召回率低于85% |
| `[独立验证发现·时序风险]` | 时序验证轨发现近2年高相关审中专利 |
| `[检索降级模式]` | patsnap_search可用但五轨未全部完成 |
| `[未完成独立检索验证]` | 检索工具完全不可用 |

|------|--------|-------------|---------|-----------|

### 步骤5：法律意见专业性审核（25分，场景自适应）

| 子项 | 标准分 | 展会场景 |
|------|--------|---------|
| C1 法律依据充分性 | 8分 | 8分 |
| C2 结论可靠性 | 8分 | 8分 |
| C3 风险缓释建议（完全缺失则0分，触发F-06） | 6分 | **8分** |
| C4 免责声明规范性 | 3分 | 3分 |

**风险规避建议自动生成（必须执行）**：高风险≥2条，中风险≥1条。

**v9.0 可执行优先级排序**：生成每条建议后，按以下矩阵计算优先级分，从高到低排序：

| 维度 | 高（3分） | 中（2分） | 低（1分） |
|------|---------|---------|---------|
| **风险消除幅度** | 完全消除侵权风险 | 大幅降低侵权概率 | 部分降低风险 |
| **可执行难度（反向）** | 易（设计改动小/成本低） | 中（需研发投入） | 难（需诉讼/谈判） |

优先级分 = 风险消除幅度分 + 可执行难度分（反向），满分6分。建议按优先级分降序列示，并注明路径类型：设计绕过 / 无效化攻击 / 许可谈判。

### 步骤6：文档完整性审核 + 综合评分与报告生成（20分，D4升级）

| 子项 | v8.1分值 | v9.0分值 | 说明 |
|------|---------|---------|------|
| D1 检索记录完整性 | 6分 | 6分 | — |
| D2 分析可追溯性 | 6分 | 6分 | — |
| D3 附录材料完备 | 5分 | 5分 | — |
| D4 版本控制规范 | **3分** | **6分** | 与F5联动：F5扣分时D4同步扣分（联动扣分上限3分） |

> D4版本控制规范提升至6分原因：版本混乱（如引用过期权利要求文本）直接导致侵权分析结论失效，危害等同于检索漏洞。F5（检索可复现性）失分时，D4自动触发联动扣分，两项合计最多扣9分，体现连锁效应。

**质量等级**：优秀90-100 / 良好80-89 / 合格70-79 / 需改进60-69 / 不合格<60 / **不可用(Fatal)**（触发一票否决）

## 召回率可视化升级（v9.0 桑基图）

在"独立检索验证"章节，除原有召回率仪表盘外，新增**四轨命中桑基图**：

```
[轨道1 语义] ──→ ┐
[轨道2 关键词] ──→ ├──→ [合并去重池 n_B] ──→ [与原报告重叠 n_C] ──→ [召回率 R_A]
[轨道3 IPC] ──→ ┤
[轨道4 申请人] ──→ ┘
                        [独立发现（不在原报告中）n_B - n_C]
```

桑基图节点宽度 = 该轨道贡献的不重复专利数，用 ECharts `sankey` 组件实现，颜色编码：
- 轨道节点：蓝色系（深浅区分四轨）
- 重叠节点：绿色（充分）/ 橙色（偏低）/ 红色（严重不足）
- 独立发现节点：红色（含高风险遗漏）/ 黄色（含中风险遗漏）/ 灰色（无风险遗漏）

## Harness 跨字段逻辑一致性规则（v9.0新增）

> 在原有结构/样式/禁用表述校验基础上，新增以下跨字段逻辑矛盾检测规则。

| 规则编号 | 检测逻辑 | 矛盾描述 |
|---------|---------|---------|
| LR-01 | A1评分 ≥ 7/8 但检索数据库数量 ≤ 1 | "高数据库覆盖分与单一数据库矛盾" |
| LR-02 | B2评分 ≥ 7/8 但高风险专利数量 = 0 | "高侵权风险判定分与零高风险专利矛盾" |
| LR-03 | C3评分 ≥ 5/6 但风险规避建议条目 = 0 | "高风险缓释分与空建议清单矛盾" |
| LR-04 | 召回率评级=充分 但独立发现专利含高风险遗漏 | "充分召回率与高风险遗漏专利矛盾" |
| LR-05 | D4评分 = 满分 但F5检索可复现性为失分项 | "满分版本控制与不可复现检索矛盾" |
| LR-06 | 综合质量等级=优秀 但任一致命缺陷被触发 | "优秀等级与致命缺陷共存矛盾" |
| LR-07 | B4评分 ≥ 4/5 但有效性分析章节无任何引证 | "高有效性分析分与无引证依据矛盾" |
| LR-08 | 报告结论=可自由实施 但高风险专利清单非空 | "FTO可实施结论与高风险专利清单矛盾" |

触发逻辑矛盾规则时，harness 输出 `logic_conflict` 类型警告，不自动修改正式报告，仅作为整改建议。

## 审核边界与免责说明

- 本技能用于FTO报告质量评审，**不替代专业法律意见**
- 独立验证结论基于智慧芽PatSnap检索数据，受限于数据库覆盖范围和检索截止日期
- FTO审核主要覆盖专利自由实施风险；除非报告另有分析，不应推定已覆盖商标、版权、商业秘密、监管合规、产品安全、合同限制等非专利风险
- 审核结果供委托方/法务团队参考，**最终审核结论由委托方确认**
- 输出报告需标注相应保密等级
- Jackknife召回率估算为统计近似值，存在抽样误差；κ修正系数基于轨道重叠比例计算，不保证精确独立性
