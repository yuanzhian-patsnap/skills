# 技术主题洞察报告 · 标准化Skill

---

## 适用场景

- 需要生成某技术领域的专利情报专题报告（HTML格式）
- 需要对现有报告进行质量诊断和迭代优化
- 需要对特定专利进行高风险排查和规避建议
- 需要综合专利、市场、论文、政策、标准多维信源输出技术战略分析

## 不适用场景

- 简单的单篇专利查询（直接用PatSnap MCP即可）
- 非HTML格式的纯文字摘要报告
- 不涉及专利分析的纯市场研究报告
- 单章节技术简报（使用 tech_report_skill 更合适）

**三条铁律：**
- ❌ **禁止自主调整章节顺序**：§0必须是第一章，§9必须是最后一章，中间章节顺序固定不变。

**违反判定**：生成结果缺少任意一章、顺序有误、或出现额外章节，均视为**生成失败**，须重新生成。

### 约束2：风格统一铁律（与历史同类型报告完全一致）

每次生成必须与历史同类型报告在以下6个维度保持完全一致，**不得因技术主题不同而自行调整风格**：

| 统一维度 | 标准要求 |
|----------|----------|
| **视觉风格** | 深色科技风CSS变量体系（`--bg-dark` / `--accent-blue` / `--accent-yellow`等），不得改为浅色或其他风格 |

| **数据标注** | 置信度双轨体系：权威来源数据（蓝色柱/正常字重）与内部估算数据（黄色柱/`.est-label`斜体）严格区分 |
| **专利风险分级** | 四级固定标识：🔴极高 / 🟠高 / 🟡中 / 🟢低，§0行动矩阵与§4高风险清单中同一专利的风险等级必须一致 |

**检查时机**：Phase 3编写完成后、Phase 4质检时，逐项对照本表确认，任意一项不符须立即修正。

## 0. 首次生成近100分保障机制

> 核心教训：报告反复修订的根本原因是"先写后查"——内容写完才发现数据无来源、HTML结构有问题。本章要求**先验后写**，并给出每项检查的具体执行方式。

### 0.1 开写前强制检查（6项，全部通过才能开始写正文）

**CHECK-1：HTML骨架验证**
```python
import re
from collections import Counter
with open("report.html", encoding="utf-8") as f:
    content = f.read()
for i in range(10):
    assert f'id="s{i}"' in content, f"缺少 #s{i} 章节锚点"
ids = re.findall(r'id="([^"]+)"', content)
dupes = {k:v for k,v in Counter(ids).items() if v>1}
assert not dupes, f"重复ID: {dupes}"
print("CHECK-1 通过")
```

## 2. 顶尖权威分析方法论框架

### 2.1 市场与竞争分析方法论

| 方法论 | 应用章节 | 使用要点 |
|--------|----------|----------|
| **Porter五力模型** | §1 | 五力均需量化锚点（市占率/集中度），颜色用描述色，不用P1紧急红 |
| **价值链分析** | §3 | 拆解上游→中游→下游，每段标注主要玩家+技术方案+近期动态 |
| **技术S曲线** | §2 | 判断技术阶段：萌芽/成长/成熟/衰退，指导布局优先级 |
| **PEST分析** | §5 | 政策/经济/社会/技术四维扫描，近期政策变化用T0标注 |
| **BCG矩阵** | §6 | 技术/产品按市场增长率×相对份额分类：明星/现金牛/问题/瘦狗 |

### 2.2 技术成熟度评估（TRL）

```
TRL 1-3：基础研究（论文为主）
TRL 4-6：技术验证（专利+样机为主）
TRL 7-9：产品化/量产（新闻+财报+招标为主）
每条技术路线均需标注当前TRL等级，并给出判断依据。
```

### 2.3 §7 前沿应用分析三步法

```
Step 1：跨域映射
  识别该技术在本领域外的潜在迁移场景
  工具：PatSnap FOS功能导向检索 + Web Search "[技术功能] application [新领域]"
  输出：3–5个潜在迁移场景清单，每个场景标注当前TRL

Step 2：TRL差距分析
  对比原场景TRL vs 迁移场景TRL，评估迁移难度
  差距≤1级：可快速迁移（重点推荐）
  差距2–3级：需中期研发投入（次优）
  差距≥4级：长期布局，当前不建议主推

Step 3：上下游联动分析
  迁移场景对上游（材料/零部件/算法）产生的新需求
  迁移场景对下游（终端产品/集成商/用户）产生的新机会
  识别迁移场景中的空白专利区（白点）→ 对应写入§6
```

## 4. §3 竞争情报三表构建规范（通用化）

> **核心原则**：§3竞争情报由三张独立表格构成，适用于任意技术主题。在开始写§3之前，先完成玩家分层（见4.1），再依次构建三表（见4.2–4.4）。

### 4.1 行业玩家分层方法（适用任意技术主题）

```
Step 1：通过PatSnap检索该技术领域专利申请量TOP10申请人
Step 2：通过Web Search识别该领域公认的国际头部企业（市占率/营收/品牌认知）
Step 3：通过行业报告/产业链图谱识别上下游关键企业
Step 4：按以下分层归类：

  国际头部玩家（表一）
  ├── 定义：在该技术领域全球专利申请量TOP或市场份额TOP，且已有商业化产品/服务
  ├── 通常数量：3–8家
  └── 来源：PatSnap专利申请人排名 + 行业市占率报告

  本土玩家（表二·时序维度）
  ├── 定义：中国大陆注册企业，在该技术领域有专利申请或量产产品
  ├── 时序范围：通常取近5年（[当前年-4] 至 [当前年]），以专利申请年或产品发布年排序
  └── 来源：PatSnap按申请人国家=CN过滤 + 产业新闻

  上下游企业（表三·价值链维度）
  ├── 上游：提供原材料/关键零部件/基础模组或核心IP的供应商
  ├── 中游：系统集成商/方案提供商/平台层
  └── 下游：终端产品制造商/应用集成商/终端用户企业
```

### 4.2 表一：国际头部玩家表（通用框架）

```
表头标准列（6列）：
| 企业名称 | 代表产品/方案 | 核心专利（经核实） | 技术路线 | 近12个月动态 | 置信度 |

填写规范：
① 企业名称：使用官方英文名，括号内注明国家/地区
② 代表产品/方案：具体到型号或版本，标注✅已量产/🔄规划中
③ 核心专利：至少1件经PatSnap核实的真实专利号，加PatSnap链接
④ 技术路线：对应§2中已定义的技术路线名称（保持一致）
⑤ 近12个月动态：标注🆕+来源+日期；无近期动态则标注"暂无公开信息"
⑥ 置信度：✅双路径核实 / ⚠️单路径待补充

待核实信息处理：
- 双路径均无法核实的信息 → 移出主表，放入独立黄色虚线"待核实框"
- 待核实框内容不得被外部引用，需标注"不可直接引用"

HTML结构要求：

## 5. 热点白点识别与高风险专利排查SOP

### 5.1 技术热点识别方法（四步法）

```
Step 1：IPC分类专利密度分析（近5年申请量，TOP3=热点，<10件=潜在白点）
Step 2：申请人集中度分析（TOP3>60%=被垄断热点）
Step 3：近期申请趋势（增速>50%=新兴热点，增速<0=技术退潮）
Step 4：论文+专利双维度验证（论文多但专利少=学术热点但商业未成熟）
```

### 5.2 技术白点识别方法（四类型）

```
IPC空白：该技术主题下某IPC子分类专利数量极少（<5件）
地域空白：国际有布局但CN无申请，或CN有布局但US/EP无申请
组合空白：A技术和B技术均有专利，但A+B组合方案无专利
时效空白：某关键专利即将到期（≤3年），进入窗口期

⚠️ 白点结论必须来自§4全量检索（见第6章6.3），三次零命中验证通过后方可写入报告。
```

### 5.3 高风险专利五步排查法

```
Step 1：PatSnap检索相关IPC分类下近5年高被引专利（被引≥10次）
Step 2：筛选权利要求涵盖"系统实现"而非"具体参数"的宽权利要求专利
Step 3：逐件进行Claim Chart比对（主权利要求各特征 vs 自身技术方案）
Step 4：四级风险定级：
  🔴极高：主权利要求每项特征均可被自身方案读入
  🟠高  ：主权利要求>50%特征可被读入
  🟡中  ：主权利要求<50%特征可被读入
  🟢低  ：权利要求需多步骤才可能读入
Step 5：§4专利清单与§0行动矩阵双向核验（🔴=P1，🟠=P2，🟡=P3）
```

### 5.4 规避建议撰写规范（三路方向）

```
每件高风险专利必须给出三路规避方向：
路线A：结构/特征绕开（修改触发侵权的具体技术特征）
路线B：功能等效替代（用不同原理实现相同功能）
路线C：授权/合作（评估是否值得主动谈判取得许可）

每路标注：★★★可行性（三星=立即可执行/二星=需验证/一星=备选）
必须包含法律声明："以上为技术分析视角，最终以FTO报告为准，建议量产前委托专利律师复核"
```

### 6.3 §4 专利全量检索五步法（核心规范）

```
【适用范围】§4 专利全景与高风险排查章节的所有专利统计、IPC分布、趋势图、白点结论
【核心原则】IPC分布图和年度趋势图须基于全量 matched_total，不得用 topk 样本替代

Step 1：IPC优先策略
  · 列出所有相关IPC分类（如 B60S1/56 / B08B3/02 / G01S13/00 等）
  · 每个IPC分类单独检索，不合并，记录各自 matched_total
  · matched_total ≤ 500 → 直接全量下载，无需分页
  · matched_total > 500 → 分段：按年份（5年/段）或按子IPC拆分，逐段全量

Step 2：关键词补充策略
  · 每个核心技术词至少设3种语言变体（中文/英文/缩写）
  · 每个技术方向至少2种关键词组合独立检索
  · 两轮检索结果取并集，去重后再进入排查流程

Step 3：申请人锁定策略
  · 发现头部申请人后，对其做申请人全量检索
  · 确保不遗漏同一申请人名义略有差异的专利

Step 4：近3年新增专项补查
  · date_from=当前年-3，date_to=当前日，单独跑一遍全量
  · 近期授权/公开专利最易被相关性排序淹没，须专项补查

Step 5：零命中验证协议
  · 若某技术方向 matched_total=0，必须换3组不同关键词二次验证
  · 三次均为0才可在报告中标注：
    「技术白点（检索日期：YYYY-MM-DD，检索策略：[说明]，三次零命中验证通过）」
  · ❌ 禁止将 topk 返回为空直接等同于全量零命中，两者语义完全不同
```

## 12. 经验总结：踩过的坑（14条）

### 🔴 坑1：竞争情报表HTML结构损坏，后续所有章节排版异常
**根本原因**：某行`<tr>`缺少`<td>`列，且全部闭合标签丢失，浏览器错误修复DOM。
**预防**：写完每行<tr>后立即验证列数；全表结尾：`</tbody></table></div>`
**首次生成操作时机**：§3三表每写完一行立即检查，不要等全表写完再验证。

### 🔴 坑2：多个重复id导致JS图表只渲染第一个
**预防**：搭建骨架时即运行CHECK-1脚本，开写前确认id唯一。

### 🔴 坑3：Python脚本replace锚点误截断，整段内容消失
**预防**：
```python
assert content.count(old_anchor) == 1, "锚点不唯一，拒绝执行替换"
assert len(new_content) > len(content) * 0.95, "文件缩减超5%，疑似截断"
```

### 🔴 坑14：仅用 topk 样本做§4全景分析，导致白点/高风险结论不可信
**症状**：§4 IPC分布图/趋势图使用 topk=20 的返回结果，或将"topk样本中未见"直接写为"技术白点"，或申请人排名仅基于样本前20件。
**危害**：整个§4结论失去统计代表性，§6白点建议和§0行动矩阵均无效；遗漏高风险专利导致FTO误判。
**预防**：§4必须执行全量检索（见第6章6.3五步法）；IPC分布图必须注明数据来源为全量matched_total，而非topk检索结果；白点结论必须附三次零命中验证记录。**报告正文（含图表标题、图例label、来源表描述、检索说明、免责声明）任意位置均禁止出现「样本」字样**，统一用「全量」「全量检索」「matched_total」替代。发布前用grep验证「样本」零命中。

### 🟠 坑4：CSS缺失导致IPC条形图不可见
**预防**：写图表前先写CSS四件套，不要等内容写完再补样式。

### 🟠 坑5：跨章节数据不一致（§0/§3/§4/§8同一专利风险等级不同）
**预防**：Phase 0建立同步表，§4写完后统一广播到§0/§3/§8，修改任一处必须同步。

### 🟠 坑6：法律判断语气绝对化（"无风险""安全"）
**预防**：§8全部写完后grep检查禁用词列表（CHECK-6），0命中才发布。

### 🟡 坑7：导航链接使用本地绝对路径，在其他设备上失效
**预防**：所有`<a href>`统一用`#s0`~`#s9`相对锚点，禁止`file:///`路径。

### 🟡 坑8：雷达图维度语义方向不一致
**预防**：雷达图所有维度统一"数值越高=需求越强"；约束类指标（功耗约束→功耗充裕度）改为正向表述。

### 🟡 坑9：待核实信息混入主表，被读者当作已核实事实引用
**预防**：待核实信息独立黄色虚线框隔离，与主表明确区分，注明"不可直接引用"。

### 🟡 坑10：技术代际时序歧义（并行发展的技术代际被误读为严格时间顺序）
**预防**：并行发展加注"代际划分以技术路线而非严格时间顺序为准"。

### 🟡 坑11：信源单一（只用专利，缺市场/新闻/标准）
**预防**：Phase 2采集时按六维信源矩阵逐类检索，每类至少有1个记录。

### 🔴 坑13：擅自调整报告章节结构（顺序/删减/新增）

**症状**：生成时认为"该主题§9不适用"而省略，或在§4后插入"补充分析"章节，或将§5与§6顺序对调。

**危害**：破坏跨章节逻辑链（§4→§0→§8三角闭环），导致读者在预期位置找不到内容，且与同系列报告风格割裂。

**预防**：生成前默读强制约束章节的"章节结构铁律"，生成后grep验证§0–§9全部存在且顺序正确；数据不足时用"待补充"占位，绝不删除章节。

### 🟡 坑12：本土时序表时间列口径混用（部分行填申请年，部分行填授权年）
**预防**：时序表时间列统一使用申请年；授权年写入"状态"列，两列互不重叠。
