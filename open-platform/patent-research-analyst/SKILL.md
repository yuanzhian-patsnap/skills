# 专利检索与分析 Agent —— 工作手册

## 一、Skill 定位

面向企业 IPR、研发工程师、科研项目负责人，针对输入的技术课题，执行完整的专利检索与分析工作流，输出结构化报告（HTML + Word 双格式）。

---

## 二、触发条件

用户输入含以下任意意图时启动本 Skill：
- 帮我做专利检索 / 专利检索分析
- 立项查新 / 科研立项
- FTO 分析 / 专利风险排查
- 技术路线分析
- 竞对专利跟踪
- 帮我分析这个技术方向的专利

---

## 三、工作流（四步全流程）

### 步骤一：信息收集

**目标**：整理课题的核心输入信息，确保检索方向准确。

收集以下字段：
1. **待解决的技术问题**【必填】：描述具体的技术难题或痛点
2. **初步的技术方案**【必填】：描述拟采用的技术手段、原理或关键技术特征
3. **关注的竞争对手**【选填】：企业名称列表（中英文均可）

**执行规则**：
- 若用户输入无法归纳出必填字段，必须继续追问，不得跳过
- 整理结果以"信息确认卡"形式输出，**等待用户明确确认后**方可进入步骤二
- 信息确认卡格式：
  ```
  📋 信息确认卡
  ─────────────────────────────
  【技术问题】xxx
  【技术方案】xxx
  【竞争对手】xxx（如未填写则标注"未指定"）
  ─────────────────────────────
  请确认以上信息是否准确？确认后将开始专利检索。
  ```

---

### 步骤二：专利检索

**目标**：构建覆盖全面、噪声低的专利检索集，作为后续分析的数据基础。

#### 2.1 检索策略

执行以下三路检索，逐步迭代：

**第一路：问题导向检索**
- 围绕"待解决的技术问题"构建检索式
- 字段优先使用 `TACD_ALL`（标题+摘要+权利要求+说明书全文）
- 部分核心关键词可结合 `TAC_ALL`（标题+摘要+权利要求）提高精准度

**第二路：方案导向检索**
- 围绕"初步技术方案"中的关键技术手段构建检索式
- 提取技术方案中的核心名词、技术特征、结构特征、功能特征作为关键词
- 尝试中英文双语关键词

**第三路：补充检索（扩展覆盖）**
- 基于前两路结果，提取：
  - **重要专利权人**（TOP 申请人）→ 用 `PA` 字段限定申请人检索
  - **高频 IPC/CPC 分类号** → 直接以分类号补充检索
  - **重要专利的引文** → 对核心专利做引文追踪（前向引用 + 后向引用）

#### 2.2 迭代优化规则

- 每轮检索后评估结果的相关性：
  - 相关性 < 60%：分析噪声来源，修改关键词或添加限定词，重新检索
  - 相关性 ≥ 60%：进入下一路检索或合并
- 最多迭代 3 轮，超出后采用当前最优检索式

#### 2.3 合并总检索式

所有检索路得到稳定检索式后，使用 `OR` 合并为**总检索式**。

**重要规则**：
- 后续所有分析均基于**总检索式的全量结果**，禁止仅取 TOP N 数据
- 记录每路检索式、检索结果数量，用于附录2

### 步骤三：五维分析

**首次进入此步骤时**，先向用户展示可分析的5个维度，让用户选择执行全部或部分：

```
📊 我可以为您提供以下5个维度的分析，请选择：
  1️⃣ 技术路线分析 — 近10年全球技术路线梳理 + 本方案定位
  2️⃣ 立项查新 — 竞对布局 + 创新空间 + 本方案新颖性
  3️⃣ FTO 风险排查 — 高风险专利清单 + Infringement Chart
  4️⃣ 竞对跟踪 — 竞对技术路线 + 近1年代表专利
  5️⃣ 近3个月新公开高风险专利
请输入数字（如"全部"或"1,3,4"）：
```

#### 分析维度1：技术路线分析

**内容**：
- 近10年全球范围内，该技术主题的主要技术路线（3-6条）
- 每条技术路线：技术手段描述、优点、缺点、代表性专利（2-3件，含专利号+标题）
- 本方案所属技术路线判定
- 本方案与其他路线的横向比较（优劣分析）

**数据来源**：总检索式全量数据 + IPC 技术构成分析 + 代表专利全文解读

**专利链接规范**：所有出现的专利号必须添加 Eureka 链接（见本文档第五节「专利链接规范」）

**输出位置**：报告第2章

---

#### 分析维度2：立项查新

包含三个子项：

**3.1 竞对布局**
- 在本方案技术路线上，已有的竞对专利布局情况
- 列出代表性专利（专利号 + 申请人 + 标题 + 技术要点）
- 所有专利号必须添加 Eureka 链接

**3.2 创新空间识别**（写 3-5 项）

从以下四个视角识别创新机会：

> **一、问题导向**
> - 技术缺陷：功能失效、性能瓶颈、可靠性不足
> - 多目标冲突：安全-成本-效率的权衡
> - 新约束条件：法规、网络安全、国产化率等
>
> **二、规避驱动**
> - 等效功能替代：架构/协议/执行机构的替代路径
> - 原理级绕行：从模拟到数字到智能化的原理跨越
> - 架构级重构：从集中式到分布式到边缘智能
>
> **三、融合创新**
> - 跨领域嫁接：引入其他行业的成熟技术
> - 多专利组合：协同布局形成专利群
> - 学科交叉：多学科交叉点
>
> **四、技术演进**
> - 新材料/新工艺
> - 数字化/智能化：AI、预测性维护、数字孪生
> - 机电软一体化

**3.3 本方案新颖性判定**
- 检索本方案的核心技术特征是否已有相同或相近披露
- 判定结论：新颖 / 部分新颖（说明哪些特征已有披露）/ 不新颖
- 列出最接近的对比文件（最多5件），所有专利号添加 Eureka 链接

**输出位置**：报告第3章（3.1/3.2/3.3）

---

#### 分析维度3：FTO 风险排查（第4章）

**4.1 高风险专利清单**

- 从总检索集中识别与本方案高度重叠的有效专利
- **排列顺序规则（强制）**：
  - 🟢 **授权有效**专利优先排列在前（法律状态为"授权有效"/"有效"/"Active"）
  - 🟡 **审中/尚未授权**专利排在后面
  - 🔴 已失效专利排在最后（一般不列入高风险清单，仅作参考）
- 表格形式（固定列顺序）：

| 序号 | 专利号（含 Eureka 链接） | 申请人 | 标题 | **法律状态** | 风险点描述 | 风险等级 |
|---|---|---|---|---|---|---|
| 1 | [CN116610973B](链接) | 哈工大 | xxx | 🟢 **授权有效** | xxx | 🔴 高 |
| 2 | [CN122087628A](链接) | 苏州热工 | xxx | 🟡 **审中（尚未授权）** | xxx | 🟠 中高 |

- **法律状态标注规范**：
  - 授权有效：`🟢 授权有效`
  - 审中/公开未授权：`🟡 审中（尚未授权）`
  - 已失效：`🔴 已失效`
- **所有专利号必须添加 Eureka 链接**（见第五节「专利链接规范」）
- 获取法律状态：调用 `mcp_patsnap-aggregation-data__detail_text` 查询每件专利的当前法律状态

**4.2 Infringement Chart**（逐件分析，基于权利要求1）

> ⚠️ **强制规则**：
> 1. 每件高风险专利的 Infringement Chart **必须且仅基于该专利的权利要求1** 进行逐特征拆解比对，不得使用其他权利要求作为比对基准。
> 2. 分析前必须调用 `mcp_patsnap-aggregation-data__detail_text` 工具获取该专利的**权利要求1完整原文**。
> 3. 表格第一列必须填写**权利要求1原文**（逐字填写，不得概括、简写或改写），每行对应权利要求1中拆解出的一个独立技术特征原文片段。
> 4. **授权状态提示规则（强制）**：
>    - 🟢 **授权有效**专利：正常进行逐特征比对，无需额外提示
>    - 🟡 **尚未授权**专利：在该专利 Infringement Chart 表格**顶部**必须添加以下灰色提示框，然后再展示比对表格：
>
>      > ⚠️ **【尚未授权提示】本专利目前处于审查阶段，尚未获得授权。以下权利要求范围仅供参考，不构成现实侵权风险。待专利授权后，需根据最终授权权利要求重新评估侵权风险。**
> 5. **分析顺序**：先分析所有授权有效专利，再分析尚未授权专利。

**制作流程**：
1. **获取法律状态**：分析前先查询该专利当前法律状态，判断是否授权有效
2. **获取原文**：调用 `mcp_patsnap-aggregation-data__detail_text` 获取专利权利要求1全文
3. **划分特征**：将权利要求1拆解为最小技术单元（技术特征），逐句/逐短语切分
4. **逐项比对**：建立"专利特征 vs 本方案特征"对照表，按全面覆盖原则逐一判定相同、等同或缺失
5. **等同判定**：对不相同特征，检验是否以基本相同手段实现相同功能、达到相同效果且无需创造性劳动即可联想到
6. **得出结论**：全部特征相同或等同 → 侵权成立；任一特征缺少或不等同 → 侵权不成立

**表格列（固定格式）**：

| 权利要求1原文 | 权利要求技术特征总结 | 本方案技术描述 | 对比结论（相同/等同/缺失） |
|---|---|---|---|
| （逐字填写权利要求1中该特征的原文，不得改写） | （用简洁语言总结该技术特征的核心） | （对应描述本方案在该特征上的实现方式） | （相同 / ⚠️等同 / ❌缺失/不等同） |
| **侵权结论** | | | ✅侵权成立 / ✅侵权不成立 + 理由 |

**输出位置**：报告第4章（4.1/4.2）

#### 分析维度4：竞对跟踪

**有指定竞对时**：
- 逐一分析各竞对在该技术问题上的专利布局
- 技术路线归属判断
- 近1年代表专利（用申请日限定，`APD: 20250101~20261231`），所有专利号添加 Eureka 链接

**未指定竞对时**：
- 基于总检索集，分别统计国内TOP3 / 海外TOP3 专利权人
- 逐一分析其技术路线
- 近1年代表专利，所有专利号添加 Eureka 链接

**输出位置**：报告第5章

---

#### 分析维度5：近3个月新公开高风险专利

- 用公开日限定近3个月（`PBD: 20260301~20260604`，随当前日期动态调整）
- 在总检索集中筛选
- 识别与本方案高度相关的专利
- 表格形式：专利号（含 Eureka 链接） | 申请人 | 公开日 | 标题 | 技术要点 | 风险描述
- **所有专利号必须添加 Eureka 链接**

**输出位置**：报告第6章

### 步骤四：报告输出

生成两种格式：**HTML 格式**（优先）+ **Word 格式**（.docx，通过 python-docx 生成）

#### 报告章节结构（7章 + 3附录）

```
第1章  项目信息收集
第2章  技术路线梳理
第3章  立项查新
  3.1 竞对布局
  3.2 创新空间
  3.3 本方案新颖性
第4章  风险排查（FTO）
  4.1 高风险专利清单（授权有效专利优先排列）
  4.2 侵权分析与结论（Infringement Chart，基于权利要求1；尚未授权专利附提示）
第5章  竞对跟踪
第6章  近3个月高相关专利
第7章  附录
  附录1：检索目标和关键词
  附录2：检索式与检索结果
        （检索目标 | 检索式 | 结果数量 | 相关性综述）
  附录3：相关专利列表
        （公开公告号（含 Eureka 链接） | 申请日 | 标题 | 摘要 | 技术问题 | 技术手段 | 技术效果）
```

#### HTML 报告规范

严格遵循全局 HTML 报告输出规范（见 EUREKA.md），包括：
- 左侧固定导航栏（220px）
- Design Token CSS 变量体系
- 7章 + 3附录均在左侧导航中有锚点
- 概览统计卡展示：检索专利总量、分析维度数、高风险专利数、竞对数量
- 专利表格、Infringement Chart 均使用标准 `.card` + `table` 样式
- **所有专利号均渲染为可点击的 Eureka 超链接**（详见第五节「专利链接规范」）
- 4.1节法律状态列使用彩色徽章样式：🟢授权有效 / 🟡审中 / 🔴已失效
- 4.2节尚未授权专利的提示框使用灰色背景 `background: #f5f5f5; border-left: 4px solid #aaa`
- 禁止引入外部 CDN、外部字体
- 禁止出现任何费用相关内容

## 四、PatSnap 工具使用规范

### 检索字段说明
| 字段 | 说明 |
|------|------|
| `TACD_ALL` | 标题+摘要+权利要求+说明书全文，用于关键词检索（主用） |
| `TAC_ALL` | 标题+摘要+权利要求，精准检索时使用 |
| `PA` | 申请人字段，用于竞对检索 |
| `IPC` | IPC 分类号 |
| `CPC` | CPC 分类号 |
| `APD` | 申请日，格式 YYYYMMDD |
| `PBD` | 公开日，格式 YYYYMMDD |

### 核心工具
- **专利搜索**：`mcp_patsnap-aggregation-data__search_patents_v3`
- **专利详情（含权利要求）**：`mcp_patsnap-aggregation-data__detail_text`
- **专利批量详情（含 patentId 查询）**：`mcp_patsnap-aggregation-data__detail_aggregation`
- **申请人排名**：`mcp_patsnap-aggregation-data__applicant_rank`
- **技术构成**：`mcp_patsnap-aggregation-data__technology_constitute`
- **申请趋势**：`mcp_patsnap-aggregation-data__trend`
- **语义搜索**：`mcp_patent-search__patsnap_search`

### 数据完整性要求
- **全量分析**：步骤三所有分析必须基于总检索式的全量结果，不得仅取 TOP N
- 如结果超过 20000 件，在附录中说明并采用分层抽样策略（按年份、IPC 分层）
- 所有引用专利必须标注来源 `[S1]` 等标记

## 五、专利链接规范（强制）

### 5.1 适用范围

报告中**所有出现专利号的位置**，包括但不限于：
- 第2章：技术路线代表专利
- 第3章：3.1竞对布局专利、3.3新颖性对比专利
- 第4章：4.1高风险清单、4.2 Infringement Chart表格
- 第5章：竞对跟踪代表专利
- 第6章：近3个月新公开专利
- 附录3：全部专利列表

### 5.2 链接构建流程（必须遵循）

**步骤1：批量查询 patentId**

在生成任何包含专利号链接的输出前，必须先调用以下工具批量获取 patentId：

```
工具：mcp_patsnap-aggregation-data__detail_aggregation
参数：patent_number = "CN122087628A,CN116610973B,CN118129820B,..."（逗号分隔，上限100件）
```

返回结果中每件专利包含 `patent_id` 字段，即 UUID 格式的 patentId。

**步骤2：构建 Eureka 详情页 URL**

URL 模板（唯一正确格式）：
```
https://eureka.zhihuiya.com/view/#/fullText'figures/?patentId={patentId}
```

示例：
```
patentId = 06a71a51-e35f-4a70-bc52-ea957193f9c1
→ URL = https://eureka.zhihuiya.com/view/#/fullText'figures/?patentId=06a71a51-e35f-4a70-bc52-ea957193f9c1
```

### 5.3 不同输出场景的链接格式

| 输出场景 | 格式规范 |
|---------|--------|
| **HTML 报告** | `<a href="https://eureka.zhihuiya.com/view/#/fullText'figures/?patentId={patentId}" target="_blank">CN122087628A</a>` |
| **Word 报告（.docx）** | 超链接形式，显示文字为专利号，URL 为详情页地址 |
| **对话 / Markdown** | `[CN122087628A](https://eureka.zhihuiya.com/view/#/fullText'figures/?patentId={patentId})` |

### 5.4 强制约束

- ✅ **必须**先调用 `mcp_patsnap-aggregation-data__detail_aggregation` 批量查询 patentId，再构建链接
- ✅ **必须**使用 `patentId`（UUID）构建详情页 URL，不得使用专利号直接拼接 URL
- ✅ 链接在 HTML 报告中必须在新标签页打开（`target="_blank"`）
- ❌ **禁止**使用 `https://eureka.zhihuiya.com/search?q=专利号` 格式（已废弃）
- ❌ **禁止**省略任何专利号的 Eureka 链接，即使已标注 `[S#]` 来源标记
- ❌ **禁止**跳过 patentId 查询步骤直接拼接 URL
- ❌ **禁止**使用伪造或未经查询的 patentId 构建链接

## 六、输出质量标准

| 维度 | 标准 |
|------|------|
| 信息收集 | 必填字段齐全，用户明确确认后方可推进 |
| 检索覆盖率 | 至少执行2路检索，合并后形成统一检索式 |
| 技术路线 | 至少识别3条技术路线，每条有代表专利 |
| FTO分析 | 每件高风险专利有完整 Infringement Chart，基于权利要求1原文 |
| FTO排序 | 4.1清单中授权有效专利排在前，尚未授权专利排在后 |
| FTO提示 | 4.2中尚未授权专利必须在表格顶部显示"尚未授权"提示框 |
| 专利链接 | 报告中所有专利号均附 Eureka 详情页链接（基于 patentId），HTML/Word/对话格式各有规范 |
| patentId 查询 | 生成任何链接前必须通过 `detail_aggregation` 查询真实 patentId，不得伪造 |
| 报告格式 | HTML 符合全局规范，附录3专利列表不少于20件 |
| 溯源性 | 所有分析结论有 `[S#]` 标注，对应附录3专利列表 |

## 七、约束与禁止事项

- ❌ 禁止在用户确认信息前自动进入检索步骤
- ❌ 禁止在步骤三中仅取 TOP N 数据做分析
- ❌ 禁止在报告中出现任何费用相关内容（代理费、官费、年费等）
- ❌ 禁止伪造专利号或检索数据
- ❌ 禁止跳过 Infringement Chart 中的等同判定步骤
- ❌ 禁止在 4.2 节使用权利要求1以外的权利要求作为比对基准
- ❌ 禁止在 4.2 节对权利要求1原文进行概括或改写，必须逐字引用
- ❌ 禁止省略任何专利号的 Eureka 链接
- ❌ 禁止对尚未授权专利省略"尚未授权"提示框
- ❌ 禁止将尚未授权专利排列在授权有效专利前面（4.1清单）
- ❌ 禁止使用 `search?q=` 格式构建 Eureka 链接（已废弃，必须使用 patentId 详情页）
- ❌ 禁止跳过 patentId 查询步骤直接拼接链接
- ✅ 每个分析结论必须有专利证据支撑
- ✅ 检索式必须记录在附录2中备查
- ✅ 4.2节每件专利分析前必须调用 `mcp_patsnap-aggregation-data__detail_text` 获取权利要求1原文
- ✅ 4.1节每件专利必须查询并标注当前法律状态
- ✅ 所有专利链接必须先通过 `mcp_patsnap-aggregation-data__detail_aggregation` 查询 patentId 后再构建
