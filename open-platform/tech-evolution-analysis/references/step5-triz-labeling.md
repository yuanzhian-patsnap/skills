# Step 5 交付报告：TRIZ 进化路线 AI 标引（v2 方法论文档）

> **流水线位置**：进化树构建 7 步法的 **Step 5（TRIZ 标引）**
> **输入**：[patent_pool_step4.json](patent_pool_step4.json) 高价值池（每条记录附 5 维子分 + 综合分）
> **输出**：[triz_labeled_pool.json](triz_labeled_pool.json) 标引完成池 + [breakthrough_index.json](breakthrough_index.json) 突破点索引
> **下游**：Step 6 四层级进化树 + 三类节点（类比/特征/空白）

## 一、v1 → v2 升级要点

| 维度 | v1 | v2 |
|------|----|----|
| 标引颗粒度 | 仅组件层（C1-C6 共 6 路） | **组件 6 路 + 关键零部件 13 路 = 19 路** |
| 子代理并发 | 12 个（每桶 1 个） | 19 个（按 v3 拆解到的零部件粒度） |
| 标引深度 | 全部一致深度 | **按 Step 4 综合分分级**（80+ 全字段精标 / 60-79 标准 / 40-59 轻量 / <40 仅入池作背景） |
| 节点定位精度 | v0/v1/v2/v3 四级粗分 | **v0-v11 全 11 节点**对齐 TRIZ 路线节点字典 |
| 一致性保障 | 单 prompt + 标准节点字典 | 单 prompt + 标准字典 + **抽查回归**（每桶 5% 由二次代理复核） |
| 输出向 Step 6 | 三层级树（整体/系统/组件） | **四层级树**（产品/大系统/子系统/组件 + 零部件细节） |

## 二、执行规约

### 2.1 标引规约

| 项目 | 实施 |
|------|------|
| 标引规约 | 沿用 [TRIZ进化路线AI标引-Prompt-v1.0.md](../TRIZ进化路线AI标引-Prompt-v1.0.md) v1.0 schema，扩展 `tier` 字段从 v3 改为 v0-v11 |
| 并发策略 | 19 个 general-purpose 子代理，按桶并行（组件级 6 桶 + 零部件级 13 桶） |
| 输入字段 | 复用 Step 4 输出的全字段：`{id, pn, title, abstract, ipc, jurisdiction, cited, family_size, value_score, applicants, score_a..e, score_total}`；论文记录为 `{id, title, text(摘要), cited, authors, org_names, doi, journal_name, pub_date}` |
| 证据约束 | 专利：综合分 ≥ 60 的项必须用 abstract + claims 作证据；< 60 项可仅用 title。**论文**：统一用 `text`（摘要）作证据（论文无 claims/IPC，摘要是唯一可靠 tier 判据） |
| 证据类型标记 | 每条记录标 `evidence_type`：`patent` / `paper`。论文证据让 Step 7 的「10 年预测」「黑天鹅哨兵」可优先取学术信号（对齐 Step 7 §2.1「10 年=论文支撑」） |
| breakthrough 定义 | 在该路线节点上属于"本域首次跃迁"或"跨域已有但本域尚未类比"标 true |

### 2.2 标引深度分级（v2 新增）

按 Step 4 综合分对 19 路标引各自做内部分级：

| 综合分区间 | 标引深度 | 资源投入 | 输出字段 |
|-----------|----------|----------|----------|
| 80-100 | **全字段精标 + 单独核验** | 子代理 1 件 1 调 + 二次代理复核 | 全字段（含 `evidence_quotes` / `tier_reasoning` / `analogy_candidate`） |
| 60-79 | **标准标引** | 子代理批量处理 | 标准字段（`dom_routes` / `tier` / `cur_node` / `breakthrough` / `note`） |
| 40-59 | **轻量标引** | 子代理批量处理 | 仅 `dom_routes + tier`（用于密度填充） |
| < 40 | **入池不标引** | 跳过标引 | 仅作 Step 7 报告统计背景 |

> 该分级使资源向真正高价值的种子专利倾斜。预估：80+ 区间约 5-10% / 60-79 约 30-40% / 40-59 约 30-40% / <40 约 20-30%（实际比例取决于 Step 4 输出分布）。

### 2.3 抽查回归（v2 新增一致性保障）

每个桶完成首轮标引后，**抽 5% 由二次代理独立复核**：

- 一致 → 通过
- 节点位置差 1（如 v3 vs v4） → 标记 `tier_drift`，取均值
- 节点位置差 ≥ 2 或 `breakthrough` 不一致 → 触发**人工 review**

> 解决 v1 中 12 个子代理独立标引时节点名称不统一（"系统自动化" vs "自动化"）的问题。同时为 v0-v11 全 11 节点的精度托底。

### 2.4 派生埋点 spawns_new_route（v3 新增 · 进化树拓扑数据源）

进化树是一片**森林**：组件长出若干并列主干路线，主干上的某个节点又会**派生**出新的子路线（"路线套路线"）。派生关系**自下而上、数据驱动**——只有在专利/文献里真的观察到"某节点上冒出了新维度的方案"才连边。这个判断**必须在标引阶段做**（标引代理手里有全文证据），Step 6 无法事后凭规则可靠推断。

**埋点判据**（满足才填 `spawns_new_route`，否则留 `[]`）：

| 判据 | 说明 |
|------|------|
| **母节点明确** | 这件专利的主方案落在某条路线 R_a 的某个节点上（= `from_route` / `from_node`） |
| **新维度出现** | 同一件专利里，除了 R_a，还明确承载了**另一条路线 R_b 的方案**，且 R_b 是在 R_a 这个成熟节点的基础上才得以展开的 |
| **因果而非并列** | R_b 的方案**依赖** R_a 节点的成果（"因为系统已经做到了 X，才能进一步做 Y"），不是两个互不相关的改进各标各的 |
| **证据可摘** | 能从原文摘出一句话支撑"此处派生" → 填 `evidence` |

> **克制同样适用**：大多数专利是单一改进，`spawns_new_route` 应为 `[]`。只有"承上启下的枢纽专利"才埋点。宁可漏埋（Step 6 有共现回退启发式兜底），不可滥埋（污染拓扑）。

> **示例**：一件"头带可根据外部环境自动调节松紧"的专利——主方案在路线 7（频率匹配 v3 均匀变化），但它**因为**实现了参数均匀可调，**才**进一步引入了路线 6（可控性 自动化）的伺服机构。→ `from_route=7, from_node=系统参数均匀变化, to_route=6, dimension=可控性提高`。

### 2.5 论文标引规则（v4 新增 · 四轨检索配套）

Step 3 v2 引入本域/跨域论文轨，论文记录走专属标引规则：

| 项目 | 论文标引实施 |
|------|-------------|
| **证据源** | 仅 `text`（摘要）。无 claims/IPC，tier 判定全靠摘要语义 |
| **tier 判定** | 用摘要描述的技术状态对照 v0-v11 字典；论文常描述**实验室验证态**，tier 往往**高于同期专利**（这正是前瞻信号的来源） |
| **evidence_type** | 固定标 `paper` |
| **breakthrough** | 论文的 breakthrough 语义是「**学术首次验证**」，不等于商用——标 true 时须在 `note` 注明"实验室验证态，未必可制造" |
| **前瞻标记** | 论文 tier 高于本域专利 max tier ≥2 时，标 `leading_signal: true` → Step 6 直接喂给空白点外推、Step 7 喂给 10 年预测/哨兵 |
| **派生埋点** | 论文同样可埋 `spawns_new_route`（学术里"因为做到 X 才能做 Y"的因果同样有效） |

> **关键价值**：论文 tier 普遍领先专利 1-3 年。当某组件出现「跨域论文已 v9 + 本域专利仅 v5」时，这个 4-tier 落差就是 Step 7 灰犀牛/10 年预测最强的结构化证据——而纯专利流水线看不到这个落差。

## 三、TRIZ 11 路线 × 节点字典（v2 标准化）

> Step 5 标引必须严格使用此字典，所有 `cur_node` / `prev_node` 字段从字典中选词。

| 路线 | 名称 | 节点字典（v0 → v11） |
|:----:|------|---------------------|
| 1 | 单-双-多系统 | 单系统 → 双系统 → 多同类系统 → 多异类系统 → 反系统 → 缩并系统 |
| 2 | 扩展裁剪 | 单功能 → 多功能 → 超系统 → 单功能极限 → 自服务 |
| 3 | 分割向微观 | 整体 → 分块 → 颗粒 → 粉末 → 胶体 → 分子 → 离子 → 原子 → 量子场 |
| 4 | 系统表面特性 | 光滑 → 粗化 → 毛细 → 多孔 → 织构 → 自适应表面 → 主动表面（执行件） |
| 5 | 系统内部结构 | 实心 → 空腔 → 多腔 → 蜂窝 → 周期晶格 → 准周期 → 拓扑结构 → 主动场 |
| 6 | 系统可控性 | 不可控 → 手动 → 半自动 → 自动 → 自适应 → AI 决策 → 自演化 |
| 7 | 系统频率匹配 | 固定 → 梯度 → 间歇 → 频率匹配 → 自适应频率 → 主动调谐 |
| 8 | 系统动态化 | 静态 → 单自由度 → 多自由度 → 柔性铰链 → 全柔性 → 流变体 |

### 3.1 路线方向语义（v4 新增 · 标引与预测的方向基准）

> **核心原则**：这 11 条路线是从**不同行业、不同技术里提炼出的公约数 —— 普遍规律**。标引的最大意义，是把一个**具体产品投影到这些普遍规律上，读出"按普遍规律它下一步该往哪走"**。
>
> **关键修正**：`tier↑` 表示**沿该路线的内在方向走得更深，不等于"更复杂"**。不同路线的方向终点不同——有的指向更结构化/更智能（增益型），有的恰恰指向**裁剪、降维、集成（裁剪型）**。预测"下一节点"时必须**沿路线自身方向**取，绝不能统一往复杂端推。

| 路线 | 内在方向 | 高 tier 端意味着 | 方向类型 |
|:----:|----------|------------------|----------|
| 1 单-双-多 | 增配 → **缩并** | 末端"缩并系统"= 用更少部件实现多系统功能 → **更简/更集成** | 先增益后裁剪 |
| 2 扩展裁剪 | 扩展 → **极限裁剪/自服务** | 末端"自服务"= 系统自己完成功能、部件趋零 → **极度精简** | **裁剪型** |
| 3 分割向微观 | 宏观 → 微观/场 | 走向分子/离子/量子场尺度 → 控制粒度更细 | 增益型 |
| 4 表面特性 | 被动 → 主动表面 | 表面从静态到可执行 | 增益型 |
| 5 内部结构 | 实心 → 主动场 | 结构从实心到拓扑/主动 | 增益型 |
| 6 可控性 | 不可控 → 自演化 | 控制智能化 | 增益型 |
| 7 频率匹配 | 固定 → 主动调谐 | 参数可调化 | 增益型 |
| 8 动态化 | 静态 → 流变体 | 刚性到柔性/流变 | 增益型 |
| 9 体几何 | 简单几何 → 仿生 | 几何效率优化（非纯复杂化，仿生常是降阻/降耗） | 效率型 |
| 10 表面几何 | 平面 → 拓扑曲面 | 曲面自由度提升 | 增益型 |
| 11 线性组合 | 单点 → 分形阵列 | 阵列维度提升 | 增益型 |

> **理想度是所有路线的共同终极方向**：理想度 = 有用功能 /（成本 + 有害作用）。增益型路线靠"增加有用功能"逼近理想度；裁剪型路线（1 末端 / 2）靠"砍成本与部件"逼近理想度。**二者殊途同归，都是进化，不存在"更复杂才更先进"。**
>
> **对成本驱动 / 降维需求的承接**：当一个产品处于成本驱动或下沉市场窗口，其主干路线天然落在**裁剪型（路线 2、路线 1 末端）**——这不是框架的例外，而是普遍规律里本就存在的方向。低端颠覆（Christensen）= 沿裁剪型路线的 tier 推进，不需要另设轴。

## 四、输出 schema（v2）

每条记录产出一份标引文档：

```json
{
  "id": "patsnap_xxxx",
  "pn": "US20180084456A1",
  "evidence_type": "patent",   // v4: patent / paper —— 论文证据供 Step 7 10 年预测/哨兵优先取用
  "leading_signal": false,     // v4: 论文 tier 领先本域专利 max ≥2 时为 true
  "score_total": 87,
  "label_depth": "full",  // full / standard / lite / skip
  "dom_routes": [
    {
      "route": 6,
      "route_name": "系统可控性",
      "tier": 6,
      "cur_node": "AI 决策",
      "prev_node": "自适应",
      "evidence_quotes": ["..."],
      "tier_reasoning": "...",
      "breakthrough": true,
      "analogy_candidate": ["C3 ANC", "C4 麦阵列"],
      "conf": "high"
    }
  ],
  "core": "...",
  "note": "...",
  "spawns_new_route": [   // v3 派生埋点（见 §4.1）
    {
      "from_route": 7,
      "from_node": "系统参数均匀变化",
      "to_route": 6,
      "dimension": "可控性提高",
      "evidence": "头带可根据外部环境自动调节松紧"
    }
  ],
  "drift_flag": false,    // v2 抽查回归标记
  "review_status": "passed"  // passed / drift / human_required
}
```

**v2 新增字段**：
- `score_total` / `label_depth`：来自 Step 4 + 本步分级
- `tier`（v0-v11 整数）+ `cur_node` / `prev_node`（节点字典词）— 替代 v1 的 `cur/prev` 自由文本
- `evidence_quotes`：高分项的原文引证（abstract/claims）
- `tier_reasoning`：tier 判定的逻辑说明（仅 full 深度填充）
- `analogy_candidate`：跨域记录可用作哪些组件的类比候选（Step 6 直接消费）
- `drift_flag` / `review_status`：抽查回归状态

**v3 新增字段（派生埋点 · 进化树拓扑的核心数据）**：
- `spawns_new_route`：**最关键的新增**。当这件专利在某条主干路线的某个节点上，**同时出现了另一个进化维度的方案**时，在此埋点。它是 Step 6 把"路线套路线"连成森林的唯一可靠数据源。
  - `from_route` / `from_node`：派生发生在哪条母路线的哪个节点（母节点 = 特征点）
  - `to_route` / `dimension`：从该节点冒出了哪条子路线 / 新维度
  - `evidence`：支撑"这里确实冒出新维度"的原文证据
  - 无派生信号时该字段为 `[]`（多数专利都是 `[]`，克制原则同样适用）

**v4 新增字段（四轨检索 · 论文前瞻信号）**：
- `evidence_type`：`patent` / `paper`。区分证据来源，Step 7 的 10 年预测与黑天鹅哨兵优先取 `paper` 证据。
- `leading_signal`：论文 tier 领先本域专利 max tier ≥2 时为 `true`。这是补偿专利滞后的核心信号——标记"实验室已验证、专利尚未跟进"的前瞻靶点。

## 五、19 路并发设计

### 5.1 子代理映射表

| 路次 | 桶名 | 父结构 | 标引焦点 |
|:---:|------|--------|----------|
| 1-6 | C1-C6 各组件 × {domain, cross} | 组件层 | 组件级演化主干（6 组件 × 2 方向 = 12 桶并发） |
| 7-19 | P1.1-P6.4 各零部件 × {domain, cross} | 零部件层 | 零部件级独立轨迹（13 零部件 × 2 方向 = 26 桶并发，部分桶可合并） |

> 实际实现按 Step 3 的 38 桶 1-1 映射成 38 个子代理；上表是**逻辑视图**。19 路是去重后的最小并发单元。

### 5.2 子代理职责

每个子代理处理一个桶，职责：

1. 读取桶内全部记录（含 Step 4 综合分）
2. 按 `label_depth` 分级处理（full / standard / lite / skip）
3. 输出 [step5_outputs/labeled_BUCKET.json](step5_outputs/) 单文件
4. 在自己处理范围内标记 `breakthrough=true` 的项
5. 提供桶内 5% 抽样列表给二次代理做回归

### 5.3 抽查回归代理

独立于上述 19 路，**1 个二次代理**接收抽样集合：

- 不读子代理的标引结果，独立标引
- 与子代理结果对比，输出 `drift_report.json`
- 触发的人工 review 项汇入 `human_review_queue.json`

## 六、Step 6 进化树构建接口（v2 四层级）

`triz_labeled_pool.json` 同时支撑**四层级**进化树：

| 层级 | 数据切分 | 节点 |
|------|----------|------|
| **L1 产品级** | 全部 6 组件 + 13 零部件汇总 | 跨大系统的整体形态跃迁信号 |
| **L2 大系统级** | 4 大系统（执行/控制/动力/传输）分组 | 子系统级特征点 |
| **L3 组件级** | 6 组件 × 11 路线 | 组件级主干 + 空白点 |
| **L4 零部件级** | 13 零部件 × 11 路线 | 零部件级独立演化轨迹（v2 新增） |

### 三类节点的标识规则（v3 重定义 · 与手绘进化树对齐）

> **v3 重大修正**：三类节点的语义全部改变。特征点从"单点价值（breakthrough）"改为"拓扑分叉（派生母节点）"；空白点从"回顾性（别人有我没有）"改为"预测性（TRIZ 外推靶点）"。

| 节点类型 | v3 定义 | 数据字段 | 本质 |
|----------|---------|----------|------|
| **特征点** | 在此节点观察到新维度方案、**派生出子路线**的拓扑分叉母节点 | `spawns_new_route[]`（标引埋点） | 树的**分叉点** |
| **类比点** | 该节点方案**可被其他路线/组件借鉴** | `analogy_candidate` + 同路线跨组件 tier 落差 ≥ 2 | 横向**可移植性** |
| **空白点** | 按 TRIZ 路线**下一步应到达、但全球无布局**的节点 | 主干 max tier + 字典外推 | **下一代预测靶点** |

> `breakthrough=true` 仍然保留标引（用于 Step 7 报告的"突破信号"统计），但**不再等于特征点**。特征点现在由 `spawns_new_route` 埋点驱动。

### Step 6 算法草案

```
for each level in {L1 产品, L2 大系统, L3 组件, L4 零部件}:
    for each unit in level:
        for each route in 1..11:
            domain_records = filter(level=unit, bucket=domain, route=R, sort by pub_date)
            cross_records  = filter(level=unit, bucket=cross,  route=R, sort by pub_date)

            # 主干：domain_records 沿时间形成 v_a → v_b → ... 主轴
            trunk = pick_main_chain(domain_records, by tier asc, by pub_date asc)

            # 空白点：跨域 max - 本域 max ≥ 2 的 tier 区间
            domain_max = max(domain_records.tier)
            cross_max  = max(cross_records.tier)
            blank_tiers = range(domain_max+1, cross_max+1) if cross_max - domain_max >= 2 else []

            # 特征点：breakthrough=true 的所有节点（合并标记）
            feature_nodes = [r for r in (domain ∪ cross) if r.breakthrough]

            # 类比点：cross_records 中 analogy_candidate 命中其他单元的项
            analogy_nodes = [r for r in cross_records if r.analogy_candidate]

            yield (level, unit, route, trunk, blank_tiers, feature_nodes, analogy_nodes)
```

## 七、本步骤局限与缓解

1. **abstract 数据成本**：综合分 ≥ 60 的项需要 abstract，意味着对入池件批量调 `patsnap_fetch`。**缓解**：Step 4 已经在筛选阶段拉取了 abstract，Step 5 直接复用，避免重复请求。
2. **节点字典的边界性**：v0-v11 字典中部分路线（如路线 3、4）的节点边界存在主观判断（"颗粒"vs"粉末"）。**缓解**：抽查回归 + 人工 review 会捕获边界争议；累积的边界判定记入字典 v2.1 注释，下次跑流水线前更新。
3. **零部件级数据稀疏**：某些零部件（如音圈、磁路）在跨域专利中可能命中很少。**缓解**：稀疏桶不强求触发空白点识别；并入父组件主干。
4. **breakthrough 判断主观性**：依赖子代理的 prompt 一致性。**缓解**：抽查回归对所有 `breakthrough=true` 项做 100% 复核（不只 5% 抽样），避免假阳性。
5. **跨域类比候选过宽**：`analogy_candidate` 字段可能过度泛化。**缓解**：Step 6 聚合时再加二次过滤（同路线 + tier 差 ≥ 2 才入选）。
