# MCP 工具调用指南

本文件定义研发创新点雷达在何时、调用哪个智慧芽MCP工具、传入什么参数。
核心原则：在正确的判断节点调用正确的工具，避免噪音检索。

---

## 总体调用策略

| 场景 | MCP调用策略 | 原因 |
|---|---|---|
| 研发周报 | **条件调用** — 三要素完整才调用 | 周报信息碎片化，不完整时检索噪音大 |
| 会议纪要 | **条件调用** — 三要素完整才调用 | 同上，口语化描述易产生误导性结果 |
| 技术方案 | **全量调用** — 每个候选点均检索 | 文档完整，检索结果可靠，价值高 |

---

## 三要素完整性判断（调用前置条件）

```
调用MCP的前置检查：
✅ 技术问题：能用一句话清晰描述
✅ 技术手段：有具体方法/结构/参数描述（非泛泛而谈）
✅ 技术效果：有可量化或可描述的改进效果

任意一项为❌ → 标记"待确认" → 输出追问清单 → 等用户补充后再调用
```

---

## 工具一：新颖性快筛检索（速度优先）

**调用时机：** 每个通过三要素完整性检查的技术点
**使用工具：** `mcp_patent-search__patsnap_search`

### 参数设置（优化版）

```json
{
  "search_strategy": ["semantic"],
  "semantic_query": "[技术问题] + [技术手段] 的完整描述句",
  "sources": ["patent"],
  "topk": 8
}
```

**速度优化说明：**
- 快筛阶段**仅用 `semantic`**，不叠加 `keyword`——减少检索耗时约 30%
- `topk` 从 10 降至 **8**——减少单轮数据传输量
- 仅当快筛结果中出现高度相似专利（需要精确比对权利要求）时，才进入工具二
- keyword 补充检索只在"新颖性评为中/低"且需要找竞争格局时按需调用

### 语义查询构造规则

将技术三要素组合成一个完整的自然语言描述句：
```
模板：[解决XX问题]，[采用XX技术手段]，[实现XX效果]
示例：解决传感器在强噪声环境下精度下降问题，
      采用在前端增加自适应滤波层的方法，
      将噪声干扰降低40%
```

### 结果解读规则

```
返回结果处理：
- 取相似度最高的前3件专利重点分析
- 检查这3件专利的权利要求，与本技术点做特征对比
- 判断标准：
  · 核心技术手段高度重合 → 新颖性低
  · 技术方向相似但实现方式不同 → 新颖性中
  · 未见高度相似方案 → 新颖性高
- 将专利号和标题列入输出报告的"参考现有技术"栏
```

## 工具二：相似专利详情获取（高风险点专用，非默认步骤）

**调用时机：** 新颖性快筛发现高度相似专利（需精确比对权利要求）时，才触发此步
**使用工具：** `mcp_patent-search__patsnap_fetch`

### 参数设置

```json
{
  "keys": ["[相似专利的专利号]"],
  "key_type": "pn",
  "module": ["basic"]
}
```

### 结果解读规则

```
获取专利全文后，重点读取：
1. 独立权利要求（claim 1）：与本技术点的核心特征逐一对比
2. 申请日：判断是否在本技术点开发之前已公开
3. 权利要求保护范围：判断本技术点是否落入其保护范围

输出：与最相似专利的逐特征对比表（简化版）
```

## 多技术点检索规则

当一份文档提炼出多个技术点时：

```
处理原则：
1. 每个技术点独立检索，不合并查询
2. 按新颖性潜力从高到低排序展示
3. 若技术点超过5个，优先检索"技术完整度=完整"的点
4. 在输出报告中明确标注每个技术点的检索是否已执行

用户提示：
"本次发现X个技术点，已对其中Y个（三要素完整）执行检索；
Z个技术点因信息不完整标记为待确认，补充后可单独触发检索。"
```

---

## 检索结果为空时的处理

```
若检索返回结果数量极少（<3件）或相关度极低：
→ 新颖性潜力标记为"高（待深度验证）"
→ 说明："当前检索未见高度相似现有技术，新颖性潜力初判为高，
          建议由IP工程师进行更系统的查新检索后再做最终判断。"
→ 不直接断言"绝对新颖"，避免误导
```
